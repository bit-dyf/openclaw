# 常设命令

常设命令为您的智能体授予特定程序的**永久操作权限**。您无需每次给出单独的任务指令，而是定义具有明确范围、触发条件和上报规则的程序——智能体将在这些边界内自主执行。

这就好比将"每周五发送周报"的口头指令升级为授权："你负责周报。每周五编写并发送，只有在出现异常时才需要上报。"

## 为什么需要常设命令？

**没有常设命令时：**

- 每项任务都需要您提示智能体
- 智能体在请求间隙处于空闲状态
- 例行工作容易被遗忘或延误
- 您成为瓶颈

**有了常设命令后：**

- 智能体在定义的边界内自主执行
- 例行工作按计划进行，无需提示
- 您只需处理例外情况和审批事项
- 智能体在空闲时间高效工作

## 工作原理

常设命令在您的[智能体工作区](/concepts/agent-workspace)文件中定义。推荐的方式是直接将其写入 `AGENTS.md`（每次会话自动注入），确保智能体始终能获取这些上下文。对于较大的配置，您也可以放在 `standing-orders.md` 等专用文件中，并在 `AGENTS.md` 中引用。

每个程序需指定：

1. **范围** — 智能体被授权执行的操作
2. **触发条件** — 何时执行（计划、事件或条件）
3. **审批门控** — 哪些操作在执行前需要人工确认
4. **上报规则** — 何时停止并寻求帮助

智能体通过工作区启动文件在每次会话中加载这些指令（参见[智能体工作区](/concepts/agent-workspace)了解自动注入文件的完整列表），并结合[定时任务](/automation/cron-jobs)执行基于时间的触发。

<Tip>
将常设命令写入 `AGENTS.md` 以确保每次会话都能加载。工作区启动程序会自动注入 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`IDENTITY.md`、`USER.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 和 `MEMORY.md`——但不会注入子目录中的任意文件。
</Tip>

## 常设命令结构

```markdown
## 程序：每周状态报告

**权限：** 汇总数据、生成报告、发送给相关方
**触发条件：** 每周五下午 4 点（通过定时任务执行）
**审批门控：** 标准报告无需审批。异常情况需标记供人工审查。
**上报：** 数据源不可用或指标异常（偏差 >2σ）时上报

### 执行步骤

1. 从配置的来源提取指标
2. 与上周数据和目标值对比
3. 在 Reports/weekly/YYYY-MM-DD.md 生成报告
4. 通过配置的渠道发送摘要
5. 将完成情况记录到 Agent/Logs/

### 禁止事项

- 不得将报告发送给外部人员
- 不得修改源数据
- 即使指标不佳也不得跳过发送——如实报告
```

## 常设命令 + 定时任务

常设命令定义智能体**被授权做什么**。[定时任务](/automation/cron-jobs)定义**何时**执行。两者协同工作：

```
常设命令："你负责每日收件箱整理"
    ↓
定时任务（每天上午 8 点）："按常设命令执行收件箱整理"
    ↓
智能体：读取常设命令 → 执行步骤 → 报告结果
```

定时任务的提示词应引用常设命令，而非重复其内容：

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+1XXXXXXXXXX" \
  --message "按常设命令执行每日收件箱整理。检查邮件中的新警报。解析、分类并持久化每个条目。向负责人报告摘要。将未知项上报。"
```

## 示例

### 示例 1：内容与社交媒体（每周循环）

```markdown
## 程序：内容与社交媒体

**权限：** 起草内容、安排发布、汇总参与度报告
**审批门控：** 前 30 天所有帖子需负责人审查，之后享有常设审批权
**触发条件：** 每周循环（周一审查 → 周中起草 → 周五简报）

### 每周循环

- **周一：** 审查平台指标和受众参与度
- **周二至周四：** 起草社交帖子，创作博客内容
- **周五：** 汇总每周营销简报 → 发送给负责人

### 内容规范

- 语气必须符合品牌风格（参见 SOUL.md 或品牌声音指南）
- 在面向公众的内容中不得以 AI 身份自我介绍
- 有数据时包含指标
- 聚焦于对受众的价值，而非自我宣传
```

### 示例 2：财务运营（事件触发）

```markdown
## 程序：财务处理

**权限：** 处理交易数据、生成报告、发送摘要
**审批门控：** 分析无需审批。建议需负责人批准。
**触发条件：** 检测到新数据文件 或 按月计划周期

### 新数据到达时

1. 在指定输入目录中检测新文件
2. 解析并分类所有交易
3. 与预算目标对比
4. 标记：异常项目、阈值突破、新增定期费用
5. 在指定输出目录生成报告
6. 通过配置的渠道向负责人发送摘要

### 上报规则

- 单笔金额 > 500 美元：立即提醒
- 某类别超出预算 20%：在报告中标记
- 无法识别的交易：向负责人询问分类
- 重试 2 次后处理失败：报告失败情况，不得猜测
```

### 示例 3：监控与告警（持续运行）

```markdown
## 程序：系统监控

**权限：** 检查系统健康状态、重启服务、发送告警
**审批门控：** 自动重启服务。重启失败两次后上报。
**触发条件：** 每次心跳周期

### 检查项

- 服务健康端点响应正常
- 磁盘空间高于阈值
- 待处理任务未过期（>24 小时）
- 投递渠道正常运行

### 响应矩阵

| 情况             | 操作                 | 是否上报                   |
| ---------------- | -------------------- | -------------------------- |
| 服务宕机         | 自动重启             | 仅当重启失败 2 次时        |
| 磁盘空间 < 10%   | 通知负责人           | 是                         |
| 任务过期 > 24h   | 提醒负责人           | 否                         |
| 渠道离线         | 记录日志并在下次重试 | 离线超过 2 小时时          |
```

## 执行-验证-报告模式

常设命令与严格的执行纪律结合时效果最佳。常设命令中的每项任务都应遵循以下循环：

1. **执行** — 完成实际工作（不要只是确认指令）
2. **验证** — 确认结果正确（文件存在、消息已投递、数据已解析）
3. **报告** — 告知负责人完成了什么以及已验证的内容

```markdown
### 执行规则

- 每项任务遵循执行-验证-报告。无例外。
- "我会去做"不是执行。做完，然后报告。
- 没有验证的"完成"不可接受。证明它。
- 执行失败时：用调整后的方案重试一次。
- 仍失败时：报告失败情况和诊断信息。绝不静默失败。
- 不得无限重试——最多 3 次，然后上报。
```

此模式可防止最常见的智能体失败模式：确认了任务却未完成。

## 多程序架构

对于管理多个关注点的智能体，将常设命令组织为具有清晰边界的独立程序：

```markdown
# 常设命令

## 程序 1：[领域 A]（每周）

...

## 程序 2：[领域 B]（每月 + 按需）

...

## 程序 3：[领域 C]（按需）

...

## 上报规则（适用于所有程序）

- [通用上报标准]
- [适用于所有程序的审批门控]
```

每个程序应包含：

- 独立的**触发频率**（每周、每月、事件驱动、持续）
- 独立的**审批门控**（某些程序需要更多监督）
- 清晰的**边界**（智能体应知道一个程序结束、另一个开始的位置）

## 最佳实践

### 应该做

- 从较窄的权限开始，随着信任的建立逐步扩展
- 为高风险操作定义明确的审批门控
- 包含"禁止事项"章节——边界与权限同样重要
- 结合定时任务实现可靠的基于时间的执行
- 每周审查智能体日志，验证常设命令是否被遵守
- 随着需求变化更新常设命令——它们是动态文档

### 应该避免

- 第一天就授予广泛权限（"做你认为最好的事"）
- 省略上报规则——每个程序都需要"何时停止并询问"的条款
- 假设智能体会记住口头指令——所有内容都写入文件
- 在单个程序中混合多个关注点——不同领域使用独立程序
- 忘记用定时任务来执行——没有触发条件的常设命令只是建议
